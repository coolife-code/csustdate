# 后端配对与解锁机制（当前实现）

本文基于当前后端代码实现整理，聚焦这几个问题：

- 每周配对怎么产生
- 用户点“解锁/跳过”后状态怎么变化
- 什么情况下会自动解锁
- 配对（Pairing）会发生什么、解锁后会发生什么

## 1. 核心对象与状态

### 1.1 Match（每周匹配记录）

`matches` 表的核心字段：

- `user1_id`, `user2_id`：本周被匹配的两位用户
- `week_key` / `week_number` / `year`：周维度标识
- `match_score`：问卷相似度分数（0-100）
- `status`：匹配互动状态

`Match.status` 枚举值：

- `pending`：刚生成匹配，双方都还没操作
- `user1_unlocked` / `user2_unlocked`：仅一方点击了“解锁”
- `both_unlocked`：双方都完成了解锁（或被系统自动双向解锁）
- `user1_skipped` / `user2_skipped`：仅一方点击了“跳过”
- `both_skipped`：双方都跳过

### 1.2 Pairing（已建立配对关系）

`pairings` 表用于表示“已互通联系方式的配对关系”：

- `match_id`：对应一条 `Match`，且唯一（一个 Match 最多一个 Pairing）
- `status`：`active` / `ended`
- `ended_at`, `ended_by`：结束时间和发起结束的用户

关键点：

- `Pairing.status` 与 `Match.status` 是两套状态
- 用户后续“结束配对”只会改 `Pairing.status=ended`，不会回写 `Match.status`

## 2. 每周配对如何生成

### 2.1 触发方式

- 定时任务：每周二 00:00 触发一次周匹配
- 兜底触发：用户访问“当前匹配”接口且本周尚未生成匹配时，会尝试执行一次匹配

### 2.2 参与匹配用户条件

当前实现中，用户需满足：

- 用户状态是 `active`
- 问卷已答题数 `>= 3`（按题目条数统计）

### 2.3 配对策略（简化说明）

- 先按偏好进行双向过滤：性别、校区、年级、学院、专业
- 若候选不足，按优先级逐级放宽过滤条件（最终至少保留性别过滤）
- 对每个候选计算问卷相似度分数，选择当前层级分数最高者进行配对
- 生成 `Match(status='pending')`

### 2.4 匹配通知

- 周匹配创建完成后，会为双方各写入一条 `match_notification` 邮件任务（队列发送）

## 3. 用户点击“解锁/跳过”后的行为

接口：

- `POST /api/matches/:matchId/unlock`
- `POST /api/matches/:matchId/skip`
- `POST /api/matches/:matchId/regret`

## 3.1 解锁（unlock）规则

1. 如果当前已经是 `both_unlocked`：直接返回成功（幂等）
2. 如果当前已是“自己跳过”或“双方跳过”：返回冲突，不能再解锁
3. 否则把状态改为：
   - 对方已解锁 -> `both_unlocked`
   - 对方未解锁 -> `userX_unlocked`
4. 若本次进入 `both_unlocked`：
   - 创建 `Pairing(status='active')`（若已存在则复用）
   - 立即给双方发“已双向解锁”邮件（此路径为直接发送，不走队列）

### 3.2 跳过（skip）规则

1. 如果当前已经 `both_unlocked`：返回冲突，不能再跳过
2. 否则把状态改为：
   - 对方已跳过 -> `both_skipped`
   - 对方未跳过 -> `userX_skipped`

### 3.3 反悔（regret）规则

1. 仅“当前用户自己处于跳过状态”时可反悔：
   - `userX_skipped`（自己是 X）
   - `both_skipped`
2. 反悔后，状态改为当前用户单边解锁：
   - 用户1反悔 -> `user1_unlocked`
   - 用户2反悔 -> `user2_unlocked`
3. 若当前已 `both_unlocked` 或当前用户并非跳过方：返回冲突

注意：

- 当前状态机不会保留“一方解锁、一方跳过”的复合状态；后一次操作会覆盖成单边 `*_unlocked` 或 `*_skipped`

## 4. 自动解锁机制（你最关心的部分）

### 4.1 触发频率

- 定时任务每 5 分钟执行一次自动解锁扫描

### 4.2 自动解锁条件

仅对满足以下全部条件的 `Match` 生效：

- `status` 属于 `pending` / `user1_unlocked` / `user2_unlocked`（即双方都没有跳过）
- `createdAt <= 当前时间 - MATCH_AUTO_UNLOCK_HOURS`
- `MATCH_AUTO_UNLOCK_HOURS` 默认值是 `5`（可通过环境变量配置）

这意味着：

- “双方都没动作”会自动解锁
- “一方已解锁、另一方未操作”也会自动解锁
- 只要出现任一 `*_skipped` / `both_skipped`，就不会进入自动解锁

### 4.3 自动解锁执行结果

对每条命中的 `Match`：

1. `Match.status` 改为 `both_unlocked`
2. 创建 `Pairing(status='active')`（幂等，已存在则不重复建）
3. 给双方各入队一条 `pairing_unlocked` 邮件任务（优先级 80，异步队列发送）

## 5. “配对会发生什么，解锁会发生什么”

### 5.1 配对（Pairing）创建后发生什么

- 代表双方已进入“可联系状态”
- 双方可在“当前配对”接口拿到对方完整资料
- 后续任一方可调用结束接口将该配对置为 `ended`

### 5.2 解锁后发生什么

- 单边解锁：仅改变 `Match.status` 为 `user1_unlocked/user2_unlocked`，不会创建 Pairing
- 双边解锁（人工或自动）：会创建 Pairing，并触发双方“联系方式开放”邮件

### 5.3 结束配对后发生什么

接口：`POST /api/pairings/:pairingId/end`

- 更新 `Pairing.status='ended'`、记录 `ended_at` 与 `ended_by`
- 不会修改历史 `Match.status`（例如仍可能是 `both_unlocked`）

## 6. 关键接口总览

匹配相关：

- `GET /api/matches/current`：当前周匹配（若本周未生成，可能触发周匹配）
- `POST /api/matches/:matchId/unlock`：解锁
- `POST /api/matches/:matchId/skip`：跳过
- `POST /api/matches/:matchId/regret`：反悔并恢复为解锁状态
- `GET /api/matches/history`：匹配历史
- `GET /api/matches/:matchId/pairing`：查该 Match 是否已生成 Pairing

配对相关：

- `GET /api/pairings/active`：当前活跃配对
- `POST /api/pairings/:pairingId/end`：结束配对
- `GET /api/pairings/history`：配对历史

## 7. 邮件发送路径（实现差异）

- 周匹配通知（`match_notification`）：入队发送
- 自动解锁后的双向解锁通知（`pairing_unlocked`）：入队发送
- 用户手动完成双向解锁后的通知：控制器内直接调用邮件服务发送（非队列）

这会导致：

- 自动解锁与手动双解锁在“邮件触发链路”上存在实现差异，但业务结果一致（双方收到联系方式开放通知）

## 8. 一句话总结

当前后端的“自动解锁”是：**只要匹配超时且双方都没人跳过（包括一方已解锁另一方未操作），系统就会升级为 `both_unlocked`，自动建 Pairing，并异步通知双方开放联系方式**。
