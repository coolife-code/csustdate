# 匹配流程详解

## 概述

DateDrop 使用稳定室友算法（Stable Roommate）进行用户配对，结合问卷向量计算合拍程度，确保配对的稳定性和准确性。

---

## 1. 用户资料包生成

每个人进来，先转成一份"资料包"。

不是分数，就是一堆信息：

- 性别：男 / 女
- 年龄、地区
- 问卷答案 → 转成一个向量（比如爱好、性格、观念）
- 之前跟谁配对过（记录一下，后面不能重复）

就这么简单，不存任何好感分。

---

## 2. 候选用户过滤

给每个人找"能配的人"——先过滤。

对每个人 A：

1. 只留异性（或他偏好的性别）
2. 踢掉上一轮配对过的人
3. 筛选问卷完成度 ≥ 50% 的用户

过滤后的候选用户数量通常远小于总用户数，这是性能优化的关键。

---

## 3. 计算合拍程度

对这剩下的候选，算"合拍程度"。

用问卷向量算：

- 爱好像不像
- 观念合不合
- 生活习惯搭不搭

算出来一个合拍分。

**注意**：只算这剩下的候选，不算所有人！这就是省时间的核心。

### 合拍分数计算公式

```
合拍分 = Σ (维度相似度 × 维度权重) × 100
```

各维度权重：
- 外貌与生活方式：15%
- 性格特征：20%
- 价值观与恋爱观：25%
- 兴趣爱好：15%
- 学业与规划：10%
- 家庭与背景：10%
- 期望偏好：5%

---

## 4. 生成偏好排序

每个人按"合拍分"排个序。

从最喜欢 → 最不喜欢排个队。

比如：
```
A 的列表：B > D > F > K > ...
```

这就是他的表白顺序。

---

## 5. 稳定室友算法

运行稳定室友算法（Stable Roommate），保证不存在两个人互相更喜欢对方而没在一起。

### 算法原理

稳定室友问题是指：给定 n 个人的偏好列表，找到一个稳定的配对方案，使得不存在两个人互相更喜欢对方却没在一起的情况。

### 算法步骤

#### 第一阶段：提案和拒绝

```
while 存在未配对的人:
    for 每个未配对的人 p:
        找到 p 的偏好列表中第一个还没拒绝过他的人 q
        
        if q 未配对:
            p 和 q 暂时配对
        else:
            r = q 当前的配对对象
            if q 更喜欢 p 而不是 r:
                q 拒绝 r
                p 和 q 配对
                r 变为未配对
            else:
                q 拒绝 p
```

#### 第二阶段：循环检测

如果第一阶段结束后还有人未配对，需要检测并处理循环。

```
while 存在循环:
    找到一个循环 (p1 → p2 → ... → pk → p1)
    其中每个 pi 的第二选择是 p(i+1)
    
    让循环中的每个人都拒绝他们的第二选择
    重新运行第一阶段
```

### 算法特性

1. **稳定性**：保证不存在"阻拦对"（blocking pair）
2. **时间复杂度**：O(n²)
3. **空间复杂度**：O(n²)
4. **不一定有解**：当 n 为奇数或偏好列表存在循环时，可能有人无法配对

### 边界情况处理

#### 奇数用户

当用户数量为奇数时，必然有一人无法配对。

处理方式：
- 随机选择一人轮空
- 或者引入"虚拟用户"填补空缺

#### 偏好列表不完整

当某些用户之间的合拍分为 0 时，偏好列表可能不完整。

处理方式：
- 只考虑合拍分 > 0 的候选
- 如果偏好列表为空，该用户本轮不参与匹配

#### 循环无解

当存在无法打破的循环时，算法可能无解。

处理方式：
- 限制最大迭代次数
- 使用贪心算法作为后备方案

## 6. 多轮匹配机制

第二轮怎么搞？很简单：

- 把本轮配对记录下来
- 下一轮每个人过滤掉自己上一轮的搭档
- 再重新算候选 → 排序 → 配对

就这么简单。

### 匹配轮次管理

```
┌─────────────────────────────────────────────┐
│              匹配轮次生命周期                 │
├─────────────────────────────────────────────┤
│                                             │
│  pending ──→ running ──→ completed          │
│     │           │                           │
│     │           └──→ cancelled              │
│     │                                       │
│     └──→ cancelled                          │
│                                             │
└─────────────────────────────────────────────┘
```

### 轮次状态说明

| 状态 | 说明 |
|------|------|
| pending | 待开始，等待定时任务触发 |
| running | 进行中，正在执行匹配算法 |
| completed | 已完成，匹配结果已生成 |
| cancelled | 已取消，异常情况终止 |

## 7. 性能优化策略

### 7.1 候选过滤优化

在计算合拍分之前先过滤，大幅减少计算量：

```
原始用户数：1000
过滤后候选数：约 100-200（取决于性别比例和历史匹配）
计算量减少：80%-90%
```

### 7.2 向量预计算

用户特征向量在问卷提交时预计算并存储，匹配时直接读取：

```
问卷提交 → 计算向量 → 存储到数据库
匹配时 → 直接读取向量 → 计算相似度
```

### 7.3 批量计算

使用批量操作减少数据库查询次数：

```javascript
// 批量获取用户向量
const vectors = await UserVector.findAll({
  where: { user_id: userIds }
})

// 批量计算相似度
const scores = calculateBatchScores(vectors)
```

### 7.4 缓存策略

对热点数据使用缓存：

- 用户向量缓存（Redis）
- 匹配历史缓存
- 偏好排序缓存

## 10. 监控与统计

### 10.1 匹配成功率

```
匹配成功率 = 双向解锁对数 / 总匹配对数
```

### 10.2 平均合拍分

```
平均合拍分 = Σ 匹配分数 / 匹配对数
```

### 10.3 用户参与度

```
参与度 = 活跃用户数 / 注册用户数
```

### 10.4 问卷完成率

```
完成率 = 完成问卷用户数 / 注册用户数
```

---

## 11. 异常处理

### 11.1 匹配失败

当匹配算法无法找到稳定解时：

1. 记录异常日志
2. 使用贪心算法作为后备
3. 通知管理员

### 11.2 数据不一致

当检测到数据不一致时：

1. 回滚当前事务
2. 记录错误日志
3. 重试匹配

### 11.3 性能超时

当匹配时间超过阈值时：

1. 终止当前匹配
2. 使用简化算法
3. 记录性能日志

## 12. 未来优化方向

### 12.1 机器学习优化

使用机器学习优化匹配算法：

- 基于历史匹配数据训练模型
- 预测匹配成功率
- 动态调整权重

### 12.2 用户反馈机制

引入用户反馈机制：

- 匹配后用户评分
- 根据反馈调整算法
- 个性化匹配策略

### 12.3 实时匹配

从每周匹配改为实时匹配：

- 用户随时可以发起匹配
- 即时返回匹配结果
- 提高用户参与度
